# Backtesting Deep-Dive Prep

Use this as the team rehearsal note for Q&A. The goal is not to sound like the backtest is perfect. The goal is to show that we understand the full process, the assumptions, the controls, the weak points, and the exact difference between a prototype validation framework and final tradable evidence.

## 1. The Safest Opening Answer

If the panel asks "Can you explain your backtesting?", start with this:

> We treat backtesting as a controlled historical experiment. First we define a hypothesis before looking at the test results. Then we collect historical prediction-market data, compute the signal only from information that would have been available at that time, split the data into train, validation, and out-of-sample test periods, apply realistic fees and position limits, and then evaluate the unseen period using risk-adjusted metrics such as Sharpe, drawdown, t-statistics, and calibration metrics. The important point is that we do not just ask "did it make money?" We ask whether the result survives costs, unseen data, robustness checks, and statistical tests.

Then add the honesty clause:

> In the current prototype, there are two layers. The Research-page strategy JSON files are illustrative scaffolding generated for the UI, so we should not overclaim those Sharpe ratios as final live empirical evidence. The trust-score and ML backtest pipeline is data-driven from local historical datasets, but it still uses proxy labels and should be described as validation evidence, not proof of deployable trading alpha.

That answer is strong because it does not hide the limitation. Professors who know backtesting usually punish overclaiming more than modest results.

## 2. Layman Explanation

Backtesting is like testing a recipe with past exam papers:

1. We write the rule first: "Only trade if the market pair is truly aligned, has enough trust, and has enough spread after costs."
2. We hide the final exam: the out-of-sample test period is not used when choosing parameters.
3. We run the rule on old data as if we were standing at each date in the past.
4. We charge realistic transaction costs, because a trade that looks profitable before fees may disappear after fees.
5. We measure both profit and pain: return, volatility, drawdown, win rate, and statistical significance.
6. We ask whether the strategy still works when assumptions are stressed.

The key message: a backtest is not a crystal ball. It is a falsification test. A good backtest tries to kill a bad strategy before real money is exposed.

## 4. Full Process From Start To End

### Step 1: Define the hypothesis

Examples:

- Cross-platform arbitrage: two venues can price the same event differently, but only true resolution-aligned pairs are tradable.
- Trust score: markets with low resolution confidence should have higher dispute or resolution-risk rates.
- Lead-lag volatility: changes in event probabilities can lead traditional volatility instruments by a few days.
- Insurance overlay: high-quality binary contracts can act as fixed-cost event insurance.

Why it matters:

- A hypothesis before the test reduces data mining.
- Professors may ask whether we searched many strategies and only showed winners. The answer is that the framework includes a negative-control strategy, mean reversion, which is deliberately shown as failing after costs.

### Step 2: Collect and normalize data

Data inputs:

- Historical Polymarket markets: `data/markets_history.parquet`
- Historical trust scores: `data/trust_scores_historical.parquet`
- Dispute events: `data/dispute_events.parquet`
- Alpha signals: `data/alpha_signals.json`
- BTC IV surface snapshot: `data/btc_iv_surface.json`
- Static strategy result JSON: `public/backtest-results/*.json`

Rerun flow:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r src_file/scripts/requirements.txt

python3 src_file/scripts/fetch_historical_markets.py --limit 5000
python3 src_file/scripts/compute_historical_trust.py
python3 src_file/scripts/fetch_dispute_events.py --source gamma
python3 src_file/scripts/join_trust_to_disputes.py
python3 src_file/scripts/run_backtest.py --sweep
python3 src_file/scripts/ml_enhanced_backtest.py --model gbm
```

What each command does:

- `fetch_historical_markets.py`: pulls resolved markets from Polymarket Gamma API.
- `compute_historical_trust.py`: computes the same trust-score logic historically, using information as of before resolution.
- `fetch_dispute_events.py`: gets dispute labels from Dune if available or from Gamma's disputed markers.
- `join_trust_to_disputes.py`: creates a joined dataset with market metadata, trust score, and dispute label.
- `run_backtest.py`: tests low-trust versus high-trust dispute rates using chi-square and logistic/AUC style checks.
- `ml_enhanced_backtest.py`: builds features, trains a time-series-aware classifier, evaluates AUC/Brier/log loss, and simulates filtering.

### Step 3: Compute signals without future information

Key principle:

- At date t, the strategy must only use information observable at or before date t.

Implemented controls:

- `src_file/backtest/strategies/base.py` states signals must not use look-ahead information.
- `src_file/backtest/engine/backtest.py` computes returns using `positions.shift(1) * asset_returns - costs`.
- That shift means today's return is earned from yesterday's position, not from a position chosen after seeing today's return.

Simple answer:

> The most important anti-look-ahead control is the one-period shift. We generate the signal first, size the position, then only earn returns from the next period. So the engine cannot accidentally trade on the same return it is trying to predict.

### Step 4: Split train, validation, and test

Configured in `src_file/backtest/config.yaml`:

- Train ends: `2025-09-30`
- Validation ends: `2026-01-31`
- Test starts: `2026-02-01`

Meaning:

- Train: learn or inspect strategy behavior.
- Validation: choose thresholds such as minimum spread or trust floor.
- Test: final unseen evaluation.

Safe answer:

> The test period is not supposed to influence design decisions. If we change parameters after seeing test performance, it is no longer a valid out-of-sample test.

### Step 5: Apply transaction costs and liquidity filters

Configured in `src_file/backtest/config.yaml`:

- Polymarket spread: 100 bps
- Kalshi spread: 150 bps
- VIX options spread: 50 bps plus commission
- ETF spread: 5 bps
- Short borrow: 80 bps per year
- Minimum market volume: USD 10,000
- Max position: 10% of capital
- Max notional per trade: USD 10,000
- Quarter-Kelly sizing where applicable
- Drawdown stop: 15%

Why this is reasonable:

- Prediction-market order books can be thin.
- A raw 1% edge is not meaningful if the combined spread is 2.5%.
- Position caps prevent one attractive-looking market from dominating the whole result.

Safe answer:

> We report performance net of costs. If a strategy only works before fees, we do not treat it as tradable.

### Step 6: Generate positions and returns

The strategy generates a signal, the risk module caps position size, the cost model charges turnover, and the engine computes returns.

Core equation:

```text
strategy_return[t] = position[t-1] * asset_return[t] - transaction_cost[t]
```

Why this matters:

- It prevents look-ahead.
- It makes trading frequency expensive.
- It turns "good idea" into "tradable after costs and risk limits".

### Step 7: Evaluate performance

The framework measures:

- Annualized return: average daily return compounded to a year.
- Annualized volatility: daily return standard deviation scaled by sqrt(252).
- Sharpe ratio: annualized excess return divided by annualized volatility.
- Sortino ratio: like Sharpe, but only penalizes downside volatility.
- Calmar ratio: annualized return divided by absolute maximum drawdown.
- Max drawdown: worst peak-to-trough loss.
- Win rate: percentage of profitable trades.
- Average win/loss ratio: whether winners are larger than losers.
- Total trades: whether the sample is large enough to be meaningful.
- t-statistic: whether average returns are large relative to noise.
- AUC-ROC: whether a classifier ranks risky markets above safe markets.
- Brier score: whether predicted probabilities are calibrated.
- Log loss: whether probability forecasts are overconfident.

Most important professor-facing point:

> We do not rely on one number. A high return with large drawdown, weak t-stat, or tiny trade count is not strong evidence.

### Step 8: Run validation checks

The validation suite includes:

- Signal monotonicity: stronger signals should produce stronger average returns.
- Factor regression: returns should not just be hidden SPY/VIX/GLD beta.
- Newey-West standard errors: adjusts t-stats for autocorrelation and heteroskedasticity.
- Robustness checks: change costs, thresholds, and liquidity assumptions.
- Event study: check returns around event windows.
- Multiple-testing discipline: t-stat >= 3.0 per Harvey, Liu and Zhu (2016) style caution.
- Bonferroni-style thresholding across six strategies.
- Walk-forward validation: rolling in-sample window, then holdout period.

Safe answer:

> The backtest is designed to answer three questions: Does the signal work in the right direction? Is it still present after costs and risk limits? Does it survive out-of-sample and robustness checks?

## 5. Strategy-by-Strategy Defense

### Strategy 1: Cross-platform arbitrage / RAAS

Hypothesis:

- Polymarket and Kalshi can price the same event differently because they have different users, legal regimes, collateral systems, and resolution mechanisms.

Signal:

```text
gross_profit = max(0, 1 - YES_poly - NO_kalshi, 1 - NO_poly - YES_kalshi)
RAAS = gross_profit * alignment_score * (1 - dispute_risk)
```

Entry rule:

- Net profit after estimated spread cost must exceed 1%.
- Alignment score must be at least 0.65.
- Trust score must be at least 60.

Why this is reasonable:

- The strategy is grounded in no-arbitrage logic.
- The alignment and trust gates prevent false arbitrage from mismatched settlement criteria.
- The code now enforces the same alignment and trust thresholds described in the methodology.

Best Q&A answer:

> Raw price gaps are not enough. We only treat a spread as a candidate trade if the pair is resolution-aligned, the trust score passes the minimum floor, and the spread survives both platforms' transaction costs.

Weakness:

- Real execution depends on order-book depth, cross-venue funding, and legal access to both venues.
- The static Research-page performance numbers are illustrative until regenerated with live matched-pair data.

### Strategy 2: Lead-lag volatility

Hypothesis:

- Prediction markets may react to event information before traditional volatility instruments fully price it.

Signal:

- Change in prediction-market probability above an 8 percentage point threshold.
- Predictive regression tests whether probability changes lead asset returns or VIX changes over horizons 1 to 5 days.

Why this is reasonable:

- Prediction markets can price specific event risk more directly than broad market instruments.
- The strategy checks horizon-by-horizon t-stats instead of assuming one fixed lag.

Best Q&A answer:

> We are not saying prediction markets always beat options markets. We are testing whether event-probability shocks contain short-horizon information for volatility. The right evidence is not just Sharpe, but predictive-regression beta and Newey-West t-statistics across 1- to 5-day horizons.

Weakness:

- Test trade count can be small.
- A lead-lag effect can decay quickly if traders notice it.
- It must be evaluated after options spreads and slippage.

### Strategy 3: Insurance overlay

Hypothesis:

- A binary prediction-market contract can behave like fixed-cost insurance for a specific event.

Why this is reasonable:

- The maximum loss is known upfront.
- For event risk, binary payoff can be cleaner than a vanilla option when the goal is occurrence coverage.

Best Q&A answer:

> This is better described as risk management than pure alpha. We test whether high-trust event binaries reduce portfolio downside or market beta when the event occurs.

Weakness:

- A binary contract covers occurrence, not severity.
- If the event is much worse than expected, the binary payout does not scale like an option payoff.
- Sample sizes are naturally small because major hedge events are rare.

### Strategy 4: Dynamic hedge

Hypothesis:

- Hedge size should increase when prediction-market probability of a risk event rises, and decrease when the event probability falls.

Why this is reasonable:

- Static hedges can be expensive because they carry protection all the time.
- Dynamic hedging tries to reduce carry cost by conditioning exposure on event probability.

Best Q&A answer:

> Dynamic hedge is an overlay. We judge it by drawdown reduction, carry-cost savings, and beta control, not by standalone alpha.

Weakness:

- Hedge timing risk.
- Transaction costs from rebalancing.
- Prediction-market probability may move after traditional markets have already adjusted.

### Strategy 5: Mean reversion

Hypothesis:

- Within-market mispricings such as YES + NO above fair value may revert.

Result interpretation:

- The sample output shows negative performance after costs.
- This is useful because it proves the framework can reject a strategy.

Best Q&A answer:

> Mean reversion is our negative control. It is included precisely because a credible backtest should show that some plausible ideas fail after fees.

Weakness:

- Small single-platform inefficiencies are often eaten by bid-ask spread.

### Strategy 6: Market making

Hypothesis:

- A market maker can earn spread while managing inventory risk.

Why it is included:

- Shows the product can model liquidity provision and inventory-sensitive strategies.

Best Q&A answer:

> Market making is simulation-only in our prototype because we do not have tick-level order-book history or queue-position data. It is an indicative microstructure model, not an executable live P&L claim.

Weakness:

- Top-of-book data does not tell us whether our orders would actually fill.
- Queue position and adverse selection dominate real market-making performance.

## 7. Key Validity Threats And How To Answer

### Look-ahead bias

Risk:

- Using future outcomes or future prices to decide today's trade.

Control:

- Historical trust score is computed as of before resolution.
- Strategy returns use shifted positions.
- Test period starts after train and validation.

Answer:

> The return calculation uses yesterday's position for today's return. That is the direct code-level guard against look-ahead bias.

### Data leakage

Risk:

- A feature accidentally contains the answer.

Control:

- Future reruns of the ML script no longer include volume in both the label and the feature set.
- Final price is used only to construct labels, not as a live prediction feature.

Answer:

> We found and fixed one leakage risk in the ML risk-label design. The clean interpretation is the rerun after that patch, not the older high AUC.

### Overfitting

Risk:

- The strategy is tuned to noise.

Control:

- Train/validation/test split.
- Parameter count limit: maximum three per strategy.
- Walk-forward validation.
- Robustness checks.
- t-stat threshold around 3.0.
- Negative-control strategy.

Answer:

> We deliberately include a losing mean-reversion strategy. That is important because if every strategy looks good, the framework is probably selecting winners rather than testing hypotheses.

### Transaction-cost optimism

Risk:

- Prediction-market spreads and slippage wipe out the edge.

Control:

- Explicit spread assumptions in config.
- Cost deducted from position changes.
- Double-cost robustness should be run before final claims.

Answer:

> The raw spread is not the edge. The edge is only what remains after bid-ask, slippage, gas, and position limits.

### Small sample size

Risk:

- Five or eight test trades are not enough for a strong conclusion.

Control:

- Show trade count.
- Use t-stat and confidence metrics.
- Treat rare-event hedging strategies as qualitative/risk-management evidence.

Answer:

> We would not call a five-trade out-of-sample result decisive. For the insurance overlay, the right claim is that the framework measures hedge behavior; the data is not enough to claim stable alpha.

### Proxy labels

Risk:

- "Resolution risk" may be approximated rather than directly observed.

Control:

- Direct dispute labels where available.
- Bad resolution states where available.
- Ambiguous final price used as a proxy for unresolved or non-convergent outcomes.

Answer:

> The label is defensible but imperfect. It measures observed resolution risk proxies, not every possible institutional loss mechanism.

### Survivorship bias

Risk:

- Only looking at markets that remain easy to fetch or resolved cleanly.

Control:

- Fetch closed markets broadly by API pagination.
- Include cancelled/disputed states where observable.
- Document missing-data limitations.

Answer:

> The dataset should include resolved and problematic markets. The remaining risk is API coverage: if the upstream API omits certain failed markets, we cannot infer them from price data alone.

## 8. Professor-Style Q&A Bank

### Q1. Are the backtest results real or simulated?

Best answer:

> The Research-page strategy JSON is illustrative scaffolding for the product UI. We should not overclaim those Sharpe ratios. The trust-score pipeline uses real local historical datasets and can be rerun, but its labels are still proxies. The correct academic claim is that we built a reproducible validation framework and have preliminary validation evidence, not that we have a production-ready trading Sharpe.

### Q2. Why should we trust a backtest at all?

Best answer:

> We should not trust a backtest blindly. We trust it only after it passes controls: no look-ahead, train/validation/test separation, realistic costs, position limits, robustness tests, and statistical significance. A backtest is a filter for rejecting weak strategies, not proof of future profit.

### Q3. How do you prevent look-ahead bias?

Best answer:

> The key implementation is `positions.shift(1)`. A signal observed at date t only affects the next return period. For trust scoring, the historical score is computed before market resolution, so the score does not know the final outcome.

### Q4. Why use Sharpe ratio?

Best answer:

> Sharpe summarizes return per unit of volatility, so it is useful for comparing strategies with different risk. But we do not rely on it alone. We also show max drawdown, Sortino, Calmar, win rate, trade count, and t-statistics because a strategy can have a decent Sharpe but still be fragile.

### Q5. Why use t-stat >= 3 instead of 2?

Best answer:

> In financial strategy testing, many signals are tried, and multiple testing makes ordinary 5% thresholds too lenient. Harvey, Liu and Zhu argue for a higher bar around 3 for expected-return discoveries. We use that as a conservative benchmark, not as magic proof.

### Q6. What if transaction costs are wrong?

Best answer:

> Then the strategy ranking changes. That is why the config makes costs explicit and why a doubled-cost robustness check is needed. Cross-platform arbitrage is especially cost-sensitive because the edge is a spread after two venues' bid-ask costs.

### Q7. Why include mean reversion if it loses?

Best answer:

> Because a serious backtest should be able to reject strategies. Mean reversion is our negative control. It demonstrates that once realistic costs are applied, some apparent prediction-market inefficiencies are not tradable.

### Q8. The ML AUC is nearly perfect. Is that believable?

Best answer:

> Not as a final number. The older result was inflated because low volume was part of the old risk label and volume was also a model feature. We patched the label design so future reruns avoid that leakage. I would not quote the old AUC as final evidence.

### Q9. Does a low trust score cause disputes?

Best answer:

> No. The claim is predictive association, not causality. Low trust score identifies markets with ambiguous criteria, weaker evidence, or higher integrity risk. Those features are plausibly associated with disputes, but the backtest does not prove causation.

### Q10. Are binary prediction-market contracts valid hedges?

Best answer:

> They can hedge event occurrence, but not event severity. A binary pays the same whether the event is mild or extreme. So prediction markets complement derivatives; they do not replace options for magnitude-sensitive hedging.

### Q11. How do you know the lead-lag effect is not spurious?

Best answer:

> We test it with predictive regressions across horizons and Newey-West standard errors to account for overlapping and autocorrelated returns. But we should still be cautious: if the out-of-sample test has few trades or weak t-statistics, we should treat it as an interesting research signal rather than production alpha.

### Q12. What would invalidate the backtesting conclusion?

Best answer:

> Three things: if the clean out-of-sample performance collapses after costs, if robustness checks show the result depends on one narrow parameter setting, or if the signal no longer works after removing leakage and proxy-label shortcuts.

### Q13. Why not just train the trust-score weights empirically?

Best answer:

> That is future work. The current 40/35/25 score is a design prior based on resolution failure modes: criteria clarity, evidence consensus, and integrity risk. Once we have enough labelled dispute outcomes, the weights should be estimated or calibrated empirically.

### Q14. Is the strategy executable in real size?

Best answer:

> Only for sufficiently liquid markets. The prototype uses USD 10,000 max notional per trade and volume filters because the long tail of prediction markets is too thin. Real deployment would require order-book depth, queue position, and cross-venue funding controls.

### Q15. What is the strongest backtesting claim you can safely make?

Best answer:

> The strongest safe claim is that we built a reproducible, cost-aware, no-look-ahead validation framework and identified which signals are plausible, which are weak, and which need cleaner reruns. We should not claim a final production Sharpe until the live data pipeline has been rerun after the leakage fix.

## 9. Short Rehearsal Script

Use this if you get 60 to 90 seconds to explain the whole thing:

> Our backtesting process starts with hypotheses rather than numbers. For example, cross-platform arbitrage is only valid when the two contracts resolve on the same event, so we first match resolution criteria and apply trust-score gates. Then we split historical data into train, validation, and out-of-sample test periods. Signals are generated using only information available at the time, positions are shifted by one period to prevent look-ahead, and transaction costs are deducted. We then evaluate the result using return, volatility, Sharpe, drawdown, t-statistics, factor regression, and robustness checks. The most important limitation is that the Research-page strategy results are illustrative UI scaffolding, while the trust-score backtest is data-driven but still depends on proxy labels. So our claim is not "we found guaranteed alpha"; our claim is "we built a disciplined framework that can test these prediction-market strategies honestly."
